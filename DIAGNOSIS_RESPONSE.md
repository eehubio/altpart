# AltPart v2.2 — GPT 诊断回应与改进说明

GPT 的诊断方向是对的，核心判断准确：**"结论显得专业，但工程证据不足以支持直接替代"**。
但它给的解法是推翻重做成企业级系统（Next.js + TypeScript + PostgreSQL + Prisma + Neo4j + 向量召回…），那是数月工程量。

本次采取**务实策略**：在现有 Vercel 原型上，先把"会误导工程师"的几个最危险问题立即修掉，同时接上真实 Gemini API。企业级重构列入路线图。

---

## 一、本次已修复（高 ROI，立即生效）

### ✅ 修复 1：取消"平均分高 = A 直接替代"（诊断 #1）

**问题**：原逻辑 `总分≥90 → A级 → Pin-to-Pin可直接替换`，但封装名一致 + 参数平均分高 ≠ 引脚兼容。

**修复**：引入**引脚证据门槛**。没有引脚映射证据时，最高只能标 `P2 Pin-to-Pin候选`，并明确提示"引脚需人工核对后方可直接替换"。只有 `pinVerified=true`（未来接入 Pin Map Engine 后）才显示 `A 直接替代`。

新等级体系：

| 等级 | 含义 | 触发条件 |
|------|------|---------|
| `A` | 直接替代 | 引脚已验证 + 高可信 |
| `P2` | Pin-to-Pin候选（待核） | 参数高匹配但引脚未验证 |
| `B` | 硬件兼容 | 可信度 ≥70 |
| `F` | 功能替代 | 可信度 ≥55 或关键参数不同 |
| `N` | 仅供新设计 | 可信度 ≥40 |
| `P0` | 数据不足 | 证据覆盖率 <40% |
| `X` | 不推荐 | 关键不兼容 |

### ✅ 修复 2：N/A 不再默认 50 分（诊断 #2）

**问题**：未知参数给 50 分，导致"大量参数未知的候选"因为未知项都拿 50 分而混进推荐。

**修复**：拆成三个独立指标——

```
技术兼容度   = 仅基于"已知"参数的加权平均（未知项完全不参与）
证据覆盖率   = 已知参数 / 应有参数（按权重）
来源可信度   = 各参数数据来源的权威度加权
        ↓
结论可信度   = 技术兼容度 × 证据覆盖率 × (0.4 + 0.6×来源可信度)
```

并加**风险封顶**：证据覆盖率<50% → 可信度≤60；仅 AI 来源 → ≤70；关键参数不匹配 → ≤55。

实测效果：一个只查到 2 个参数（其余 N/A）的候选，即使已知参数全满分，结论可信度仅 21 分 → 直接淘汰，不再混入推荐。

### ✅ 修复 3：单位归一化（诊断 #5）

**问题**：原 `parseNum()` 只取字符串第一个数字，错误处理 `2.0 to 3.6 V`、`500 µV` vs `1 mV`、`±15 V`、`-40 to 125 °C`。

**修复**：新增 `units.js`，把原始值解析为规范化 Quantity `{min, typ, max, unit, dim, isRange}`：
- SI 前缀正确换算（µV/mV/V/kV、KB/MB、MHz/GHz…）
- 范围识别（`a to b` / `a~b` / `±x` / 负温度范围）
- 同量纲才比较，跨量纲不误判
- 温度/电压范围用**覆盖度**判断（候选范围是否包住原型号）

实测：`500µV` 与 `1mV` 正确识别为同量纲不同值（接近度 0.7）；候选 `-40~125°C` 正确判定**完全覆盖**原 `-40~85°C`。

### ✅ 修复 4：AI 数据显著降权（诊断 #6）

**问题**：AI 搜索结果统一标 `confidence=medium`，被当成中等可信证据。

**修复**：建立数据源可信度权重表——

```
ezplm/人工确认  1.0    数据手册 0.95
分销商API       0.8-0.85
AI搜索/推断     0.45   ← 显著降权
```

AI 来源的参数会拉低"来源可信度"，进而压低"结论可信度"，并触发"≤70 封顶"。前端明确标注 `AI搜索 ⚠`。

### ✅ 修复 5：接上真实 Gemini API

`gemini.js` 重写：
- 模型可配置（`GEMINI_MODEL` 环境变量，默认 `gemini-2.0-flash`）
- **重试机制**：429/503 限流 → 指数退避重试 3 次
- **超时控制**：25s AbortController，避免 Serverless 挂死
- **search 降级**：若模型不支持 google_search grounding，自动去掉重试
- **temperature=0.2**：降低幻觉
- prompt 强化："严禁编造，找不到填 N/A"

## 二、前端同步更新

## 三、暂未做（列入路线图，需要更大工程投入）

GPT 诊断中这些方向是对的，但属于"企业级重构"，建议作为后续阶段：

| 诊断项 | 为何暂缓 | 路线图阶段 |
|--------|---------|-----------|
| #3 按品类拆 Rule Pack | 当前 7 品类共用容差表够用；拆分是优化非急救 | P1 |
| #4 稳定 Parameter Code | 需重构整个数据层 schema | P1 |
| #7 混合检索替代 AI 召回 | 需 100万器件库 + Neo4j + 向量库 | P2 |
| Pin Map Engine | 需引脚级 datasheet 解析能力 | P1 |
| Next.js/TS/PostgreSQL 重构 | 数月工程，当前原型阶段不必要 | P0(工程化) |
| ezPLM Provider 抽象 | 当前 adapter 够用，Provider 是规模化才需要 | P2 |
| 持久化（KV/PG）替代内存 | 演示阶段内存够用 | P0(工程化) |

**关键判断**：当前最该先做扎实的是**运算放大器品类**（参数维度丰富、替代逻辑可验证），而非 MCU（Pin Map 和软件兼容太复杂）。本次单位归一化和证据模型已为此打好基础。

## 四、接 Gemini 后如何验证

1. Vercel 控制台配置 `GEMINI_API_KEY`（必填）
2. 访问 `https://你的域名/api/health`，确认 `geminiConfigured: true` 和模型名
3. 首页输入真实型号（如 `LM358`），观察：
   - 数据路径徽章（本地库命中 / 联网搜索）
   - 每个候选的"结论可信度四件套"
   - AI 来源的候选可信度被合理压低
   - 引脚未验证的不会标"直接替代"

实测建议先用 `gemini-2.0-flash`（稳定、支持搜索、便宜）。若需更强推理可切 `gemini-2.5-flash`。
